# Extraction Matrix — SAASTools formalization -> Elegy burden-of-proof reset

Scope: workflow formalization artifacts already changed or planned in SAASTools (workflow models/sharing, WS3 governance scripts, agent-orchestration primitives).

Decision rule: do not move code into Elegy unless it clearly proves why shared code is needed at all. Every candidate should be classified as one of these lanes:

- `.NET authority` — canonical governed semantics, schemas, fixtures, or validation that other runtimes must consume exactly
- `Rust runtime` — self-contained shared executable/runtime capability
- `consumer-local` — host-specific runtime behavior that should stay in SAASTools or another app repo
- `artifacts/docs` — schemas, policy files, workflow definitions, or docs that do not need a shared library

Topology note: the current Elegy repo is package-family based. There is no catch-all `src/Elegy.Formalization/...` source tree. Extraction targets in this matrix therefore map to the real package families under `src/`, the Rust runtime family under `rust/`, or consumer-local ownership when no shared Elegy code surface is justified.

## Execution ordering (recommended)
1. Move real authority artifacts first: workflow models, share/export DTOs, and policy artifacts/scripts that clearly need a single source of truth.
2. Rebuild the migration matrix continuously around burden-of-proof dispositions instead of assuming every helper deserves a shared package.
3. Keep host-specific orchestration, dispatch, endpoints, and DI local unless at least two real consumers prove a stronger shared boundary.
4. Consume only the Elegy surfaces that survive the authority/runtime reset: canonical `.NET` contracts and the Rust runtime/operator stack.
