# Spec Workflow

## Create A New Spec When

- behavior should be described before or during implementation
- acceptance criteria need to be explicit
- multiple contributors need the same intended behavior contract

## Update An Existing Spec When

- the behavior slice is still the same feature or boundary
- acceptance criteria evolved but the same spec still applies
- validation commands or links need to be refreshed

## Compact Spec Fields

- `title`
- `status`
- `owner`
- `problem`
- `goals`
- `non-goals`
- `behavior`
- `acceptance criteria`
- `validation`
- `links`

## Filename Rule

Use `slug.md`.

## Status Values

- `draft`
- `active`
- `completed`
- `superseded`
- `deprecated`
